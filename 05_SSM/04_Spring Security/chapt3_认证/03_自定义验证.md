# 1. 自定义验证

有些时候，我们的数据库可能并不会像SpringSecurity默认的那样进行设计，而是采用自定义的表结构，这种情况下，上面两种方式就很难进行验证了，此时我们得编写自定义验证，来应对各种任意变化的情况。

## 1.1 实现UserDetailsService接口

既然需要自定义，那么我们就需要**自行实现UserDetailsService**或是<u>功能更完善的UserDetailsManager接口</u>，这里为了简单，我们直接选择前者进行实现：

```java
@Service   //需要在WebConfiguration开启包扫描
public class AuthorizeService implements UserDetailsService {

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        return null;
    }
}
```

现在我们需要去实现这个`loadUserByUsername`方法，表示在验证的时候通过自定义的方式，根据给定的用户名查询用户，并封装为`UserDetails`对象返回，然后由SpringSecurity将我们返回的对象与用户登录的信息进行核验，基本流程实际上跟之前是一样的，只是现在由我们自己来提供用户查询方式。

## 1.3 编写查询逻辑与完善Service

接着我们自行编写对应的查询操作，首先创建一个对应的实体类：

```java
@Data
public class Account {
    int id;
    String username;
    String password;
}
```

然后是根据用户名查询用户的Mapper接口：

```java
public interface UserMapper {
    @Select("select * from user where username = #{username}")
    Account findUserByName(String username);
}
```

最后我们在配置类上添加相应的包扫描：

```java
@EnableWebMvc
@Configuration
@ComponentScans({
        @ComponentScan("com.example.controller"),
        @ComponentScan("com.example.service")
})
@MapperScan("com.example.mapper")
public class WebConfiguration implements WebMvcConfigurer {
    ...
}
```

然后我们来到Service这边进行一下完善，从数据库中进行查询：

```java
@Service
public class AuthorizeService implements UserDetailsService {

    @Resource
    UserMapper mapper;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        Account account = mapper.findUserByName(username);
        if(account == null)
            throw new UsernameNotFoundException("用户名或密码错误");
        return User
                .withUsername(username)
                .password(account.getPassword())
                .build();
    }
}
```

这样，我们就通过自定义的方式实现了数据库信息查询，并完成用户登录操作。


## 1.4 实际开发中的自定义登录接口与密文校验避坑

在前后端分离并结合 JWT 的项目中，我们往往不会使用 Spring Security 默认的表单登录，而是自己编写 `UserController` 来处理 `/login` 登录逻辑。在这种情况下，极易在使用 MyBatis/MyBatis-Plus 查询数据库时踩坑。

### 常见错误：明文密码直接查库

很多时候我们会习惯性地写出如下查询代码：

```java
// 错误写法
QueryWrapper<Account> wrapper = new QueryWrapper<>();
wrapper.eq("username", username).eq("password", password);
Account user = userMapper.selectOne(wrapper);
```

**这是完全错误的**。因为前面我们配置了 `BCryptPasswordEncoder`，数据库中存储的密码应当是加密后的密文，而前端传入的是明文，并且 **BCrypt 算法每次加密的盐值不同，密文也会不同**，直接用 `eq` 查库永远匹配不上。

### 正确流程：先查用户，再比对密码

正确的自定义登录接口逻辑必须遵循以下三步：
1. **只按用户名查库**：获取到包含密文密码的用户实体对象。
2. **密码比对交由 PasswordEncoder**：使用 `passwordEncoder.matches(明文, 密文)` 进行校验，不要自己写 `equals` 比较。
3. **发放 Token**：校验通过后再生成并返回 JWT。

正确的 Controller 代码结构示例如下：

```java
@RestController
@RequestMapping("/api/user")
public class UserController {

    @Resource
    private UserMapper userMapper;
    
    @Resource
    private PasswordEncoder passwordEncoder;

    @PostMapping("/login")
    public Result login(@RequestBody LoginDTO loginData) {
        String username = loginData.getUsername();
        String rawPassword = loginData.getPassword(); // 前端传来的明文密码

        // 1. 仅通过用户名查询数据库
        QueryWrapper<Account> wrapper = new QueryWrapper<>();
        wrapper.eq("username", username);
        Account user = userMapper.selectOne(wrapper);

        // 2. 判断用户是否存在
        if (user == null) {
            return Result.error("用户名或密码错误");
        }

        // 3. 使用 matches 方法比对明文与数据库中的密文
        if (!passwordEncoder.matches(rawPassword, user.getPassword())) {
            return Result.error("用户名或密码错误");
        }

        // 4. 校验通过，生成 JWT 等后续操作
        String token = JwtUtil.createToken(user.getUsername());
        return Result.success(token);
    }
}
```

### 补充注意点
* **注册/新增用户时**：入库前必须使用 `passwordEncoder.encode(rawPassword)` 将明文加密为密文后再执行 `insert`。
* **历史数据兼容**：如果数据库表中已经存在明文密码，改为密文校验后会导致历史用户全部登录失败，需要通过脚本或手动将历史密码全部重新 Hash 加密一遍。
