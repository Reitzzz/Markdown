之前我们已经详细了解了Java 9到Java 17的各项新特性。Java 17作为一个重要的LTS（长期支持）版本，已经被大量企业和各大主流框架（如Spring Boot 3.x）广泛采用。

但Java的进化并未停止。2023年9月，Java 21正式发布，这又是一个极其重磅的**LTS长期维护版本**！它不仅补全了之前版本中一些实验性语法的最后一块拼图，还带来了能够改变高并发编程格局的“虚拟线程”。

接下来，我们就来梳理一下Java 18到Java 21中，那些你必须掌握的核心新特性。

## 1.1 Java 18-20 过渡版本特性

Java 18到20依然是每半年发布一次的非LTS版本，主要目的是为了孵化和预览Java 21的大招，不过其中也有几个实用的改动值得注意：

**1. 默认字符集指定为 UTF-8 (Java 18)**

在Java 18之前，Java程序的默认字符集是依赖于操作系统的。比如在Windows中文版下，默认字符集通常是GBK，这就导致了我们在跨平台部署或者读取文件时，经常遇到令人头疼的乱码惨案。
从Java 18开始，官方终于拍板：无论你是什么操作系统，Java的默认字符集统统改为UTF-8。

**2. 简易 Web 服务器 (Java 18)**

类似于Python的 `python -m http.server`，Java 18引入了一个开箱即用的静态Web服务器。
环境配置好后，只需在命令行输入：
```bash
jwebserver
```
就可以在当前目录下启动一个默认端口为8000的静态文件服务器，这在进行前端测试或者临时共享文件时非常方便。

## 1.2 史诗级更新：虚拟线程 (Virtual Threads)

这是Java 21中最具革命性的特性（在Java 19和20中经历了两次预览，终于在21正式转正）。

在传统的并发编程中，我们使用 `Thread` 类创建的线程，底层都会一对一映射到操作系统的内核线程上。操作系统线程非常笨重，创建和销毁的代价极大，极其消耗内存。这就导致了我们在一台机器上最多只能创建几千个活跃线程，成为高并发的瓶颈。

而Java 21带来的**虚拟线程（Virtual Threads）**是由JVM在用户态自行管理的轻量级线程。你可以轻松创建上百万个虚拟线程，而不会导致系统崩溃。

```java
public static void main(String[] args) {
    // 之前我们创建线程池
    // ExecutorService oldExecutor = Executors.newFixedThreadPool(10);
    
    // 现在我们可以直接创建一个为每个任务开启一个虚拟线程的执行器
    try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
        for (int i = 0; i < 100000; i++) {   // 一口气提交十万个任务！
            final int taskId = i;
            executor.submit(() -> {
                
                System.out.println("任务 " + taskId + " 运行在: " + Thread.currentThread()); 
                // 打印当前执行的线程，你会发现它被标记为 VirtualThread

                
                try {
                    Thread.sleep(1000);  // 即使在虚拟线程中阻塞，也不会阻塞底层的系统线程
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        }
    } // try-with-resources 结束时会自动等待所有虚拟线程执行完毕
}
```

有了虚拟线程，我们再也不用为了追求高并发而去写那些晦涩难懂的异步非阻塞代码（响应式编程）了。直接用最传统的同步阻塞写法，交给虚拟线程去跑，性能就能直接起飞。

## 1.3 顺序集合 (Sequenced Collections)

回顾一下我们之前用过的集合类。如果我问你：怎么获取一个 `List` 的第一个和最后一个元素？
之前我们的写法通常是这样的：
```java
List<String> list = new ArrayList<>(Arrays.asList("A", "B", "C"));
String first = list.get(0);
String last = list.get(list.size() - 1);  // 写起来略显啰嗦
```

如果是 `LinkedHashSet` 呢？获取最后一个元素更麻烦，因为 `Set` 连 `get` 方法都没有！

为了解决这种集合访问缺乏统一标准的问题，Java 21引入了 `SequencedCollection`、`SequencedSet` 和 `SequencedMap` 接口。对于所有具有明确“相遇顺序”的集合，现在都拥有了一套统一且直观的API：

```java
public static void main(String[] args) {
    List<String> list = new ArrayList<>(List.of("B", "C"));
    
    list.addFirst("A"); // 直接在头部添加
    list.addLast("D");  // 直接在尾部添加
    
    System.out.println(list.getFirst()); // 输出 A
    System.out.println(list.getLast());  // 输出 D
    
    // 甚至可以直接获取一个反转视图，且不改变原集合的数据
    List<String> reversed = list.reversed();
    System.out.println(reversed); // 输出 [D, C, B, A]
}
```

## 1.4 Switch 模式匹配 (转正)

还记得我们在Java 14/17中学的增强版 `switch` 表达式吗？在Java 21中，它迎来了终极形态——支持**类型匹配**。

以前我们判断一个对象的具体类型，只能用一堆 `if-else` 配合 `instanceof`：
```java
public static void formatter(Object obj) {
    if (obj instanceof Integer i) {
        System.out.println("这是一个整数: " + i);
    } else if (obj instanceof String s) {
        System.out.println("这是一个字符串，长度为: " + s.length());
    } else if (obj == null) {
        System.out.println("这是个null");
    } else {
        System.out.println("其他未知类型");
    }
}
```

现在，这一切都可以塞进 `switch` 里面，并且可以直接对 `null` 进行处理：

```java
public static void formatter(Object obj) {
    String result = switch (obj) {
        case Integer i -> String.format("这是一个整数: %d", i);
        case String s -> String.format("这是一个字符串，长度为: %d", s.length());
        // 甚至可以在 case 后面追加 when 子句进行条件过滤（类似于断言）
        case Double d when d > 0 -> "这是一个正浮点数"; 
        case null -> "这是个null";
        default -> "其他未知类型";
    };
    System.out.println(result);
}
```
结合我们在Java 17学过的“密封类型（sealed classes）”，如果在 `switch` 中穷举了密封类的所有子类，连 `default` 分支都可以省略，极大地提高了代码的安全性。

## 1.5 Record 模式匹配 (转正)

最后，我们来看看针对Java 14引入的“记录类型（Record）”的究极优化。

在使用 `instanceof` 判断Record类型时，我们之前判断完还需要调用 `getter` 才能拿到里面的数据：
```java
// 假设我们有一个记录类型
public record User(String name, int age) {}

public static void test(Object obj) {
    if (obj instanceof User user) {
        // 判断完之后，还要手动去拿属性
        System.out.println("名字: " + user.name() + ", 年龄: " + user.age());
    }
}
```

在Java 21中，配合Record模式，我们在匹配的同时就可以直接将对象的属性解构提取出来，一步到位：

```java
public static void test(Object obj) {
    // 直接在 instanceof 后面把 Record 内部的属性拆解出来定义成变量
    if (obj instanceof User(String name, int age)) {
        // 下面直接使用解构出来的 name 和 age 变量即可
        System.out.println("名字: " + name + ", 年龄: " + age);
    }
}
```

这种解构写法同样可以完美融入到前面提到的 `switch` 模式匹配中，可谓是强强联手，让Java的语法也拥有了不逊色于现代新兴语言的简洁度。
