# 1. 持久化

## 1.1 持久化背景与方案

我们知道，Redis数据库中的数据都是存放在内存中，虽然很高效，但是这样存在一个非常严重的问题，如果突然停电，那我们的数据不就全部丢失了吗？它不像硬盘上的数据，断电依然能够保存。

这个时候我们就需要持久化，我们需要将我们的数据备份到硬盘上，防止断电或是机器故障导致的数据丢失。

持久化的实现方式有两种方案：一种是直接保存当前**已经存储的数据** (RDB) ，相当于复制内存中的数据到硬盘上，需要恢复数据时直接读取即可；还有一种就是保存我们存放数据的**所有过程** (AOF) ，需要恢复数据时，只需要将整个过程完整地重演一遍就能保证与之前数据库中的内容一致。

## 1.2 RDB (Redis Database)

### 1.2.1 概念与定义

RDB 的全称是 **Redis Database**。在技术语境中，它通常被翻译为 **Redis 数据快照**。

它就像是给 Redis 内存里的数据在某一个特定的时间点拍了一张“照片”。它将这一瞬间所有的键值对状态完整地记录下来，并压缩保存为硬盘上的二进制文件。

### 1.2.2 RDB 保存命令

RDB就是我们所说的第一种解决方案，那么如何将数据保存到本地呢？我们可以使用命令：

```sql
save  --SAVE：先停再存
--停：主线程立刻停止处理所有客户端发来的读写请求（也就是阻塞）。
--存：全心全意把内存数据写入硬盘的 dump.rdb 文件。
--继续：等全部写完之后，才重新开始处理客户端的请求

bgsave --BGSAVE：边做边存

--分身（极短停顿）：主线程花几毫秒的时间，利用操作系统的 fork 机制“克隆”出一个子进程。
--边做边存（并行执行）：
--子进程（存）：拿着刚才克隆时的“数据快照”，默默在后台往硬盘里写数据。
--主线程（做）：主线程根本不管写硬盘的事，继续以极快的速度接收和处理客户端的读写请求。两者互不干扰。
--结束：子进程写完文件后，自己默默退出，主线程依然在正常工作

```

### 1.2.3 RDB 文件生成与恢复

执行后，会在服务端目录下生成一个dump.rdb文件，而这个文件中就保存了内存中存放的数据，当服务器重启后，**会自动加载里面的内容到对应数据库中**。保存后我们可以关闭服务器：

```sql
shutdown
```

重启后可以看到数据依然存在。

### 1.2.4 RDB 自动保存配置

虽然这种方式非常方便，但是由于会完整复制所有的数据，如果数据库中的数据量比较大，那么复制一次可能就需要花费大量的时间，所以我们可以每隔一段时间自动进行保存；还有就是，如果我们基本上都是在进行读操作，而没有进行写操作，实际上只需要偶尔保存一次即可，因为数据几乎没有怎么变化，可能两次保存的都是一样的数据。

我们可以在配置文件**redis.conf**中设置自动保存，并设定在一段时间内写入多少数据时，执行一次保存操作：

```sql
save 300 10     --300秒（5分钟）内有10个写入
save 60 10000   --60秒（1分钟）内有10000个写入
```

**配置的save使用的都是bgsave后台执行**。
