# 缓存雪崩

![img](https://s2.loli.net/2023/03/06/BwncOXbRU4WFgkV.png)

当**你的Redis服务器炸了或是大量的Key在同一时间过期**，这时相当于缓存直接GG了，那么**如果这时又有很多的请求来访问不同的数据，同一时间内缓存服务器就得向数据库大量发起请求来重新建立缓存，很容易把数据库也搞GG**。

这种情况属于 *“全面崩盘”*，可以形象地比喻为：**整面城墙突然大面积坍塌**（大量Key失效或Redis宕机），外面的敌军从四面八方冲进城内；或者是防洪大坝全线溃堤，洪水直接淹没了下游的数据库。

## 解决方案

针对缓存雪崩，核心思路是避免集中失效以及保证缓存层和后端数据库的生存能力：

1. **搭建Redis集群（提升高可用性）**：这是防范Redis单机宕机引发雪崩的最根本办法。通过搭建集群（如引入哨兵模式 Sentinel 或搭建 Redis Cluster），即使主节点挂掉，备用节点也能迅速顶上，防止单点故障导致整个缓存层瘫痪。
2. **过期时间打散（加随机值）**：在给大量数据设置缓存过期时间时，绝对不能让它们在同一秒过期。可以在原有的基础过期时间上，额外加上一个随机的时间范围（比如 1~5分钟），让缓存的失效时间均匀分布。
3. **服务熔断与降级限流**：如果缓存已经雪崩，为了保护数据库不被打死，可以在微服务网关层开启限流，直接丢弃部分请求；或者触发熔断降级机制，直接给前端返回默认的数据或“系统繁忙”提示。牺牲部分用户的体验来保全整个系统的核心安全。
