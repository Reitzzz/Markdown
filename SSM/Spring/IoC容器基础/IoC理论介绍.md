# IoC理论介绍

在我们之前的图书管理系统 Web 应用程序中，整个程序依靠各个部分相互协作来共同完成操作。例如，要展示借阅信息列表，流程如下：
1. 首先使用 Servlet 进行请求和响应的数据处理。
2. 请求的数据全部交给对应的 Service（业务层）来处理。
3. 当 Service 发现要从数据库中获取数据时，再向对应的 Mapper 发起请求。

它们之间就像连接在一起的齿轮，谁也离不开谁。就像一个团队，每个人的分工都很明确，流水线上的一套操作必须环环相扣，这是一种**高度耦合**的体系。

## 传统高耦合架构的痛点

虽然这样的体系逻辑非常清晰，整个流程也能够让人快速了解，但是存在一个很严重的问题：

在我们现在的软件项目高速迭代时代，App 经常需要更新，什么功能当下最火就马不停蹄地跟进开发。这就很容易出现一种情况——之前写好的代码、实现的功能，需要全部推翻改成新的功能。此时我们不得不去修改某些流水线上的模块，但这会直接导致**整个流水线的引用关系大面积更新**。

比如下面的情况：

```java
class A {
    private List<B> list;
    public B test(B b) {
        return null;
    }
}

class C {
    public C(B b) {}
}

class B { }
```

可以看到，`A` 和 `C` 都在大量地直接使用 `B`。但是某一天，`B` 的实现已经过时了，此时来了一个功能实现得更好的 `D`，我们需要用这个新的类来完成业务。

因为类之间的关联性太强了，此时会开始大面积报错。所有之前用了 `B` 的类，都得挨个进行修改，全部替换成 `D`，这简直是灾难！包括我们之前编写的实战项目，如果我们不想用某个 Service 实现类了，想使用其他的实现类用不同的逻辑做这些功能，只能每个类都去挨个修改。当项目特别庞大时，光是改个类名导致的连带修改就足够耗费大量时间。

因此，**高耦合度**带来的缺点是很明显的，也是现代软件开发中很致命的问题。

## 引入 IoC 理论解耦



如果要改善高耦合的情况，我们只能将各个模块进行解耦，让各个模块之间的依赖性不再那么强。也就是说，Service 的实现类不再由我们硬编码决定，而是**让程序自己决定**。所有的实现类对象全部交给程序来管理，所有对象之间的关系也由程序来动态决定，这就引入了 IoC 理论。

**IOC (Inversion of Control)**，翻译为：**“控制反转”**。
它把复杂系统分解成相互合作的对象，这些对象类通过封装以后，内部实现对外部是透明的，从而降低了解决问题的复杂度，而且可以灵活地被重用和扩展。

我们可以将对象交给 **IoC 容器**进行管理。比如当我们需要一个接口的实现时，由容器根据配置文件来决定到底给我们哪一个实现类。这样，我们就不用再关心要去使用哪一个具体的实现类了，只需要关心：给到我的一定是一个可以正常使用的实现类。接口定义了什么，我们就只管调用。

这样，视图层和业务层的开发人员就可以完全解耦，大幅提升开发效率。

## IoC 容器加持下的代码结构

还是之前的场景，但在有了 IoC 容器加持之后，代码变成了面向接口编程：

```java
public static void main(String[] args) {
    A a = new A();
    // 假设 IoC 是一个容器类。具体实现由容器动态提供。
    // 比如现在在 IoC 容器中管理的 Service 实现是 B，那么我们拿到的就是 B
    a.test(IoC.getBean(Service.class));   
}

class A {
    private List<Service> list;   // 一律使用 Service 接口，具体实现由 IoC 容器提供
    public Service test(Service b){
        return null;
    }
}

interface Service { }   // 使用 Service 做一个顶层抽象

class B implements Service {}  // B 是具体实现类，并交给 IoC 容器管理
```

当具体实现类发生修改时，我们同样只需要将新的实现类交给 IoC 容器管理，而无需修改之前的任何核心业务代码：

```java
interface Service { }

class D implements Service {}   // 现在实现类变成了 D，但是之前使用 Service 接口的代码并不会报错
```

## 总结

即使底层实现类发生了修改，也不会导致与其相关联的类出现错误从而需要大面积修改。通过**定义抽象 + 容器管理**的形式，我们就可以将原有的强关联解除。

**高内聚，低耦合**，是现代软件开发的设计目标。而 Spring 框架就为我们提供了这样一个 IoC 容器来进行对象的管理。一个由 Spring IoC 容器实例化、组装和管理的对象，我们称之为 **Bean**。
